# Плагин `longreads` — страница задания

**URL:** `/longreads/...` (страница конкретного задания/лонгрида)

Набор скриптов для улучшения работы на странице отдельного задания.

---

## Файлы и их назначение

### `homework_weight_fix.js`

Отображает вес задания в итоговой оценке курса.

**Как работает:**

1. Извлекает `longreadId` из URL (`/longreads/123`).
2. Запрашивает `/api/micro-lms/longreads/{id}/materials?limit=10000`.
3. Ищет в ответе поле `estimation.activity.weight`.
4. Ждёт появления `ul.task-info` в DOM.
5. Клонирует элемент «Оценка» из списка метаинформации, меняет в нём иконку и текст на «Вес: XX%» и вставляет рядом.

Также содержит логику для отображения количества штрафных дней (late days), если они применимы к заданию.

**Взаимодействие со страницей:**

- Читает DOM: `ul.task-info`, `.task-info__item`
- Вставляет: новый `<li>` с весом задания
- Fetch: `/api/micro-lms/longreads/{id}/materials`

### `instant_doc_view_fix.js`

Позволяет просматривать вложенные файлы и изображения в модальном окне прямо на странице задания.

**Как работает:**

1. Создаёт скрытый модальный оверлей (`<div class="cu-lms-modal-overlay">`) с `<img>` или `<iframe>` внутри.
2. `MutationObserver` отслеживает появление вложений (ссылок с атрибутом `download`, изображений, ссылок на документы).
3. При клике на вложение перехватывает навигацию (`e.preventDefault()`), открывает модальное окно и загружает файл внутрь него.
4. Закрытие — по клику на оверлей или кнопку ✕.
5. Анимация появления через CSS `opacity` и `transform: scale`.

**Взаимодействие со страницей:**

- Вставляет: `<div>` модального окна в `document.body`, `<style>` анимации в `<head>`
- Перехватывает: клики по вложениям задания
- Fetch-запросов не делает

### `rename_hw.js`

Автоматически переименовывает файл перед загрузкой на сервер по заданному шаблону.

**Как работает:**

1. Читает настройки из `browser.storage.sync`: флаг `autoRenameEnabled`, шаблон `autoRenameTemplate`.
2. Перехватывает `change`-событие на `<input type="file">` через `MutationObserver` (ждёт появления поля).
3. Получает имя и фамилию студента из API (`/api/account/me`) и кэширует.
4. Определяет номер недели:
   - Сначала из элементов навигации (текст «неделя N»)
   - Затем из заголовков `<h4>` (текст «ДЗ N»)
   - В крайнем случае — вычисляет по последней дате из истории задачи
5. Создаёт `File` объект с новым именем из шаблона (например: `Иванов_неделя5.pdf`) через `DataTransfer`.
6. Подставляет переименованный файл в `input.files`.

**Поддерживаемые шаблоны:**

- `dz_fi` → `dz_Фамилия_Имя_неделяN.ext`
- `fi_dz` → `Фамилия_Имя_dz_N.ext`
- и другие, заданные в попапе

**Взаимодействие со страницей:**

- Читает DOM: `<input type="file">`, навигация курса, история задач
- Модифицирует: `input.files` (через DataTransfer API)
- Fetch: `/api/account/me`

### `task_status_adaptation.js`

Отображает кастомные статусы («Метод скипа», «Доработка») прямо на странице задания.

**Как работает:**

1. Загружает список задач студента с API (`/api/micro-lms/tasks/student`) один раз и кэширует.
2. `MutationObserver` ждёт появления элементов `cu-student-task` на странице.
3. Для каждого задания:
   - Проверяет, входит ли оно в `localStorage`-сет скипнутых задач (`cu.lms.skipped-tasks`).
   - Проверяет API-данные: если задача отклонена и дата rejectAt позже submitAt — значит это «Доработка».
4. Заменяет или дополняет стандартный чип статуса (`tui-chip`) кастомным.

**Взаимодействие со страницей:**

- Читает DOM: `cu-student-task`, `.task-name`, `tui-chip`
- Модифицирует: текст и стиль чипа статуса
- Fetch: `/api/micro-lms/tasks/student`
- Хранилище: `localStorage` (cu.lms.skipped-tasks)
